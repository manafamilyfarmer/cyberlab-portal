## Learned
- On a slower Proxmox host, a full clone can run right up against CAP_CLONE (600s) — and if it *times out* mid-clone, the error-path destroy races the still-locked clone and leaves an ORPHANED VM (stopped, with disk). That orphan is reaper-immune (persistent-box name lacks the `b2-` prefix), so it must be destroyed manually and it blocks the next reservation of that vmid ("config file already exists"). Fix: size CAP_CLONE with real headroom over measured clone time so the clone never times out in the first place.
- Never `pkill` an in-flight provision task — it skips the error-path teardown and leaves a dangling PENDING lab + reservation + lease. Let it reach a terminal state (or reconcile the residue by hand with release_reservation).
- CAP_IP_APPLIED (guest-agent IP confirm) scales with host I/O load: fine at 300s for early boxes, but later boxes booting against contended storage need more. Batch provisioning wants a generous confirm cap.
- Persistent per-student boxes are reaper-safe because they always hold an ACTIVE DB RESERVATION — that alone is the protector. v1 ALSO gated on the `b2-` name prefix, which had the OPPOSITE failure mode: a FAILED student clone left a `s<NN>-`-named orphan that matched no prefix and was therefore un-reapable, wedging the next reservation of that vmid. v2 removes the name gate and keys purely on reservation state: real boxes are protected by their reservation, orphans (no reservation) are always reapable. Reap-by-identity-attribute (name) is a trap; reap-by-authoritative-state (reservation) is correct.
- An orphan whose age can't be derived (task log rotated out, no config ctime) must NOT be skipped forever — "unknown age → skip" lets it linger indefinitely and keep blocking its vmid. Stamp a first-seen timestamp on first sighting and reap once that stamp ages past grace; bounded at ~2 grace windows.
- Reserve-then-clone still races a stale orphan CONFIG the qemu list doesn't surface (the 9006 case), so the allocator can hand out a vmid Proxmox still half-holds and the clone fails "config file already exists". Provisioning must SELF-HEAL: reap the blocking orphan (guarded) and retry the next free vmid, bounded — never hard-fail and never wedge.
- Provisioning allocation must be ATOMIC (reserve-then-clone). Check-then-act ("lowest free number" then clone) races under concurrent/retried provisions — two tasks pick the same VMID/IP. The DB (unique constraint + row locks) must arbitrate.
- Reserving the VMID first also makes the error path reliably reap residue (it knows the exact vmid even on a clone timeout).
- Parallel full clones contend for storage I/O and run slower — bound the clone wait with headroom (CAP_CLONE=600s), not the single-clone cap.
- Wazuh stores ALERTS by default, not raw events — a rule match is what surfaces in alerts.json. To archive every ingested event (not just matches), enable logall_json on the manager. So a quiet alerts.json does NOT mean the agent isn't forwarding; confirm at the archive/ingestion layer.
